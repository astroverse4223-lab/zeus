# Zeus Knowledge Base (Local RAG / File-Drop Memory) — Design

**Date:** 2026-06-11
**Status:** Approved (design); pending implementation plan
**Scope:** Feature 1 of 2 (RAG first, then Auto-update — independent specs)

## 1. Goal

Let the user drag files or folders into Zeus, index them locally, and have Zeus
answer questions over those documents. Fully local and private — embeddings are
generated by the user's own Ollama instance; document text never leaves the
machine.

## 2. Decisions (locked)

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Embeddings | **Ollama, local** (`nomic-embed-text`, 768-dim) | Private, free, offline. Ollama already integrated. |
| Retrieval | **Tool the model calls** (`knowledge_search`) | No token cost when irrelevant; fits existing tool registry; model controls when to search. |
| File types | **Text/Markdown/code + PDF + folders (recursive)** | Covers the common "my docs" cases. No `.docx` in v1. |
| Vector storage | **Hybrid: JSON metadata + `vectors.bin` Float32 sidecar** | Fast load, low memory, no native deps (keeps portable build clean). |
| Indexing trigger | **Manual** (index on drop; manual re-index button) | YAGNI — no file-watching in v1. |
| UI | **Dedicated Knowledge panel** + global window drag-drop | Discoverable, matches existing settings/panel patterns. |

## 3. Architecture

New self-contained module `electron/knowledge/`, plumbed into the existing
tool-registry → IPC → preload → renderer patterns. No new app-wide architecture.

```
Renderer (Knowledge panel)            Main process (electron/knowledge/)
  drag/drop, Add files/folder  ──IPC──▶  index.cjs (orchestrator)
  source list + delete/reindex            ├── extractors.cjs  (path → text)
  Ollama/model status                     ├── chunker.cjs     (text → chunks)
                                          ├── embedder.cjs    (chunks → vectors via Ollama)
  Chat → model calls                      └── store.cjs       (persist + cosine top-k)
    knowledge_search tool  ──IPC/tool──▶  index.search(query, k)
```

### 3.1 Module units

Each unit has one purpose, a small interface, and is testable in isolation.

- **`extractors.cjs`** — `extractText(filePath) → string` and
  `expandPaths(paths) → filePath[]`.
  - Dispatch by extension: UTF-8 read for `.txt .md .markdown .json .csv` and
    common code extensions; `pdf-parse` for `.pdf`.
  - Folder input: recursive walk, skipping binaries and `node_modules`, `.git`,
    `dist`, `release`, and files over a size cap (e.g. 5 MB).
  - Unsupported/binary files are skipped with a reported reason.
  - **Depends on:** `pdf-parse` (new dep), `fs`.

- **`chunker.cjs`** — `chunk(text, { size≈800, overlap≈150 }) → { text, pos }[]`.
  - Splits on sentence/line boundaries where possible; never mid-word.
  - Pure function. **Depends on:** nothing.

- **`embedder.cjs`** — `embed(texts[]) → Float32Array[]`.
  - Calls Ollama `POST http://localhost:11434/api/embeddings` with
    `model: 'nomic-embed-text'`, one request per text (batched sequentially with
    progress callback).
  - Throws typed errors: `OLLAMA_DOWN` (connection refused) and
    `MODEL_MISSING` (model not pulled) so callers can show actionable messages.
  - **Depends on:** Ollama HTTP endpoint (reuses existing localhost:11434 helper).

- **`store.cjs`** — persistence + search.
  - `load()` / `save()` — reads/writes `index.json` + `vectors.bin`.
  - `addChunks(sourceId, chunks, vectors)`, `removeSource(sourceId)`.
  - `search(queryVector, k) → { score, text, sourceName, pos }[]` via
    brute-force cosine over the Float32 vector array.
  - **Depends on:** `fs`.

- **`index.cjs`** — orchestrator tying the above together.
  - `ingest(paths, onProgress) → { added, skipped }` —
    expand → extract → chunk → embed → store, emitting progress per file.
  - `removeSource(id)`, `listSources()`, `stats()`, `search(queryText, k)`.

### 3.2 Data model

`%APPDATA%\zeus-ai\knowledge\index.json`:

```jsonc
{
  "version": 1,
  "model": "nomic-embed-text",
  "dim": 768,
  "sources": [
    { "id": "uuid", "path": "C:\\...\\manual.pdf", "name": "manual.pdf",
      "type": "pdf", "addedAt": "2026-06-11T00:00:00.000Z", "chunkCount": 42 }
  ],
  "chunks": [
    { "id": "uuid", "sourceId": "uuid", "pos": 0, "text": "chunk text…",
      "vecOffset": 0 }
  ]
}
```

`%APPDATA%\zeus-ai\knowledge\vectors.bin`: concatenated `Float32` vectors,
`dim` floats each. A chunk's vector lives at `vecOffset * dim`. On `removeSource`
the file is rewritten compactly (sources are few and removal is rare).

### 3.3 IPC + preload surface

Main-process handlers (registered in `main.cjs`):

| Channel | Type | Purpose |
|---------|------|---------|
| `zeus:kb-add` | invoke `(paths)` | Ingest; streams `zeus:kb-progress` events |
| `zeus:kb-progress` | event | `{ file, done, total, phase }` (mirrors Ollama-pull pattern) |
| `zeus:kb-list` | invoke | List sources with chunk counts |
| `zeus:kb-remove` | invoke `(sourceId)` | Remove a source + its vectors |
| `zeus:kb-stats` | invoke | `{ sourceCount, chunkCount, model, ollamaReady }` |

`preload.cjs` exposes on `window.zeus`: `kbAdd`, `kbList`, `kbRemove`,
`kbStats`, `onKbProgress`.

### 3.4 The `knowledge_search` tool

Registered in the existing tools array in `main.cjs`:

```
name: 'knowledge_search'
description: Search the user's indexed personal documents/knowledge base for
             relevant passages. Use when the user asks about their own files,
             notes, manuals, or previously provided documents.
schema: { query: string, k?: number (default 5) }
```

Handler: `index.search(query, k)` → returns matched passages formatted with
source name + text so the model can cite them. If the KB is empty or Ollama is
down, returns a clear message (not an error) so the model can tell the user.

## 4. UI — Knowledge panel

New renderer component (reached from the sidebar/settings, consistent with the
existing panels):

- **Drop zone** + "Add files" / "Add folder" buttons (folder uses the existing
  `pickDirectory` IPC; files use a new file-picker IPC or HTML input).
- **Source list**: name, type icon, chunk count, added date, delete button.
- **Status row**: Ollama running? `nomic-embed-text` pulled? With a one-click
  hint/link to pull it if missing (reuses existing Ollama pull flow).
- **Progress**: per-file ingest progress bar driven by `zeus:kb-progress`.
- **Re-index** action per source (re-extract + re-embed).
- **Global drag-drop**: dropping files anywhere on the app window routes to
  `kbAdd` for fast capture.

## 5. Error handling

- **Ollama down / model missing** — `embedder` throws typed errors; ingest
  surfaces them in the UI status row, and `knowledge_search` returns an
  actionable message ("Start Ollama and run `ollama pull nomic-embed-text`").
- **Unsupported/oversized/binary files** — skipped and reported in the ingest
  result summary, never crash the batch.
- **Corrupt/missing index files** — `store.load()` treats a missing or unparsable
  index as empty and rebuilds on next save (no hard failure on startup).
- **Partial ingest failure** — a file that fails extraction is skipped; already
  embedded chunks are persisted so progress isn't lost.

## 6. Testing strategy

Pure units get unit tests; the Ollama boundary is mocked.

- `chunker` — size/overlap, boundary handling, tiny/empty input.
- `store` — add/remove round-trip, cosine top-k ordering, vector offset
  integrity after a `removeSource` rewrite, corrupt-file recovery.
- `extractors` — extension dispatch, folder walk skip rules, size cap, binary
  rejection. (PDF extraction tested against a small fixture.)
- `embedder` — typed-error mapping (`OLLAMA_DOWN`, `MODEL_MISSING`) against a
  mocked HTTP endpoint.
- `index.search` — end-to-end with a mocked embedder and a seeded store.

## 7. Out of scope (v1 / future)

- File-watching / auto re-index on change.
- `.docx` and other office formats.
- OpenAI/bundled embedding backends (Ollama-only for v1).
- Per-conversation "use my docs" toggle or always-inject mode.
- Cross-encoder re-ranking; multi-KB / collections.

## 8. New dependencies

- `pdf-parse` (PDF text extraction). No native build step.

## 9. Affected existing files

- `electron/main.cjs` — register IPC handlers + `knowledge_search` tool.
- `electron/preload.cjs` — expose `kb*` methods + `onKbProgress`.
- `src/` — new Knowledge panel component + entry point in sidebar/settings;
  global window drag-drop handler.
- `package.json` — add `pdf-parse`.
